Domain: Autonomy vs. Guardrails · Human-in-the-Loop Escalation Design

Scenario

An asset manager runs a Claude-based "Trade Order Review Agent" that reads incoming order tickets and decides whether a trade can be routed straight to execution or must be escalated to a human trader for sign-off. Early production data shows the agent escalates almost everything — including routine, low-risk trades — because the team's only mechanism for controlling autonomy is a single line in the system prompt: "Use your judgment about which trades are risky enough to need human review." Traders are now buried in escalations, and the efficiency case for the agent is falling apart. Separately, compliance is worried about the opposite failure: a large, unusual trade slipping through as "low risk" because the model misjudged it.

Efficiency SLA: No more than 15% of trades may be escalated to a human.
Risk SLA: 100% of trades above a defined notional/volatility threshold must be escalated — zero tolerance.
Auditability: Every escalation/no-escalation decision must be traceable to a specific, inspectable reason — not "the model felt it was fine."

Question

Which redesign of the escalation logic best satisfies all three constraints?

A. Rewrite the system prompt with a longer, more detailed description of what "risky" means, including several worked examples of risky vs. safe trades, so the model's judgment improves.
B. Insert a deterministic pre-check layer before the model even sees the ticket: a rules engine evaluates notional size, volatility band, and counterparty flags against fixed thresholds. Trades that trip any threshold are auto-escalated without model involvement. For trades under all thresholds, the model makes a routing recommendation, which is logged with its stated reasoning, but the model is never the one deciding whether a threshold-breaching trade escalates.
C. Have the model call an "escalate_trade" tool whenever it judges a trade risky, and simply trust that this tool call is the correct signal, since it's now expressed as structured tool use rather than free text.
D. Set escalation to happen automatically for every trade above a small fixed dollar amount (e.g., $10,000), regardless of any other factor, since a single simple rule is easy for compliance to understand.

Answer key

Correct: B. A deterministic rules engine gates the zero-tolerance threshold before the model is ever consulted, so the risk SLA is structurally guaranteed rather than dependent on model judgment. The model only exercises discretion on the sub-threshold population, which is exactly where a bit of judgment is appropriate and where the 15% escalation ceiling becomes achievable, since it's no longer escalating things a fixed rule already caught. Logging the model's stated reasoning for each recommendation gives compliance an inspectable trail without making the model the final arbiter of the risk-critical cases.

Why A fails: more detailed instructions and examples still leave "risky enough" as a judgment call the model makes fresh each time — it can reduce over-escalation somewhat, but can't drive the zero-tolerance risk SLA to true zero, since it's still probabilistic.

Why C fails: wrapping the same judgment in a tool call doesn't change what's deciding — it's still the model's discretion, just expressed structurally. Structured tool use fixes *format*, not *authority over a zero-tolerance decision*.

Why D fails: a single fixed-dollar threshold ignores volatility and counterparty risk entirely, so it will both escalate harmless large-but-safe trades (blowing the efficiency SLA) and fail to catch small-notional-but-high-volatility trades that are genuinely risky (blowing the risk SLA) — simplicity here doesn't map to the actual risk surface.
